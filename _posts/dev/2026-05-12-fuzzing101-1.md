# Fuzzing101 Exercise 1 — Xpdf로 배우는 퍼징 입문
 
- **타겟**: Xpdf 3.02  
- **목표**: AFL++로 퍼징 환경을 구축하고, 실제 취약점(CVE-2019-13288)을 발견 및 분석한다  
- **CVE**: CVE-2019-13288

---

## 퍼징(Fuzzing)이란?

퍼징은 프로그램에 **이상한 입력을 마구 집어넣어서 프로그램이 터지는지(crash) 확인하는 테스트 기법**이다.

예를 들어 PDF 뷰어에 정상 PDF 대신 글자를 랜덤으로 바꾼 이상한 파일을 넣어보는 것이다. 프로그램이 이 입력을 제대로 처리 못 하면 `Segmentation fault`(비정상 종료)가 발생하고, 이게 곧 **보안 취약점**이 될 수 있다.

이번에 사용한 **AFL++(American Fuzzy Lop++)** 는 단순히 랜덤하게 입력을 바꾸는 게 아니라, **코드 커버리지(어떤 코드가 실행됐는지)** 를 추적해서 더 많은 코드 경로를 탐색하도록 똑똑하게 입력을 변형한다.

```
일반 퍼저:  그냥 랜덤 입력 → 운에 맡김
AFL++:      입력 변형 → 코드 경로 추적 → 새 경로 발견 시 저장 → 반복
```

---

## CVE-2019-13288이란?

**CVE-2019-13288**은 Xpdf 3.02의 `Parser.cc` 파일에 존재하는 취약점이다.

조작된(crafted) PDF 파일을 통해 `Parser::getObj` 함수가 **무한 재귀(infinite recursion)** 를 일으키도록 유발할 수 있다.

### 왜 위험한가?

프로그램에서 함수가 호출될 때마다 **스택(stack)** 에 **스택 프레임(stack frame)** 이 쌓인다. 스택 프레임에는 함수의 지역 변수, 복귀 주소 등이 저장된다.

```
함수 A 호출 → 스택에 프레임 추가
  함수 B 호출 → 스택에 프레임 추가
    함수 B 호출 → 스택에 프레임 추가  (재귀!)
      함수 B 호출 → 스택에 프레임 추가  (재귀!)
        ...반복...
          💥 스택 메모리 고갈 → Segmentation fault
```

함수가 재귀적으로 너무 많이 호출되면 스택 메모리가 고갈되어 프로그램이 강제 종료된다. 이를 **스택 오버플로우(Stack Overflow)** 라고 한다.

그 결과, 원격 공격자는 조작된 PDF 파일 하나로 Xpdf를 강제 종료시키는 **서비스 거부(DoS, Denial of Service) 공격**을 수행할 수 있다.

취약점이 트리거될 경우 GDB backtrace에서 `Parser::getObj`가 수십~수백 번 반복되는 것을 볼 수 있다:

```
#0   Parser::getObj (...)  at Parser.cc:XXX
#1   Parser::getObj (...)  at Parser.cc:XXX
#2   Parser::getObj (...)  at Parser.cc:XXX
...  (수백 번 반복)
```

| 항목 | 내용 |
|------|------|
| **영향 버전** | Xpdf 3.02 |
| **취약 파일** | `xpdf/Parser.cc` |
| **취약 함수** | `Parser::getObj` |
| **버그 유형** | 무한 재귀 → 스택 오버플로우 |
| **영향** | 서비스 거부(DoS) |
| **CVSS 점수** | 6.5 (Medium) |

## 환경 구축

### 1. 타겟 다운로드 및 일반 빌드

```bash
cd $HOME
mkdir fuzzing_xpdf && cd fuzzing_xpdf

# Xpdf 3.02 다운로드
wget https://dl.xpdfreader.com/old/xpdf-3.02.tar.gz
tar -xvzf xpdf-3.02.tar.gz

# 빌드에 필요한 도구 설치 후 빌드
cd xpdf-3.02
sudo apt update && sudo apt install -y build-essential gcc
./configure --prefix="$HOME/fuzzing_xpdf/install/"
make
make install
```

**명령어 설명**
- `wget [URL]` : 인터넷에서 파일을 다운로드한다
- `tar -xvzf [파일]` : `.tar.gz` 압축 파일을 푼다. `-x`는 압축 해제, `-v`는 진행상황 출력, `-z`는 gzip 형식, `-f`는 파일 지정
- `./configure --prefix=[경로]` : 빌드 전 환경 설정 단계. `--prefix`는 프로그램이 설치될 경로를 지정한다
- `make` : 소스코드를 컴파일해서 실행 파일을 만든다
- `make install` : 컴파일된 파일을 `--prefix`로 지정한 경로에 설치한다

**빌드(Build)** 란 소스코드(사람이 읽는 코드)를 컴퓨터가 실행할 수 있는 프로그램으로 변환하는 과정이다.

### 3. AFL++ 설치

#### 의존성 설치

```bash
sudo apt-get update
sudo apt-get install -y build-essential python3-dev automake git flex bison \
  libglib2.0-dev libpixman-1-dev python3-setuptools

# clang-11 설치 시도, 없으면 최신 버전으로 fallback
sudo apt-get install -y lld-11 llvm-11 llvm-11-dev clang-11 \
  || sudo apt-get install -y lld llvm llvm-dev clang

sudo apt-get install -y gcc-$(gcc --version|head -n1|sed 's/.* //'|sed 's/\..*//')-plugin-dev \
  libstdc++-$(gcc --version|head -n1|sed 's/.* //'|sed 's/\..*//')-dev
```

**명령어 설명**
- `||` : 앞 명령어가 실패하면 뒤 명령어를 실행한다. Ubuntu 24.04에서는 clang-11이 저장소에서 삭제되어 자동으로 `clang`(최신 버전)이 설치된다
- `$(명령어)` : 괄호 안 명령어의 실행 결과를 문자열로 삽입한다. gcc 버전 번호를 자동으로 가져오는 데 사용한다

**왜 clang-18이 설치되나?**  
- 가이드는 clang-11 기준으로 작성됐지만, Ubuntu 22.04 이후 저장소에서 clang-11이 삭제됐다. `||` 뒤의 버전 없는 `clang`을 설치하면 현재 최신 버전인 **clang-18**이 자동으로 설치된다.

#### AFL++ 소스 빌드

```bash
cd $HOME
git clone https://github.com/AFLplusplus/AFLplusplus && cd AFLplusplus
export LLVM_CONFIG="llvm-config-11"
make distrib
sudo make install
```

**명령어 설명**
- `git clone [URL]` : 원격 저장소의 코드를 내 컴퓨터로 복사한다
- `export LLVM_CONFIG="llvm-config-11"` : 환경 변수를 설정한다. AFL++ 빌드 시 어떤 LLVM을 사용할지 지정한다
- `make distrib` : AFL++의 모든 컴포넌트를 빌드한다

`make distrib` 실행 시 아래와 같은 오류가 나올 수 있지만 `(ignored)` 표시가 붙어 있으므로 무시해도 된다:

```
ERROR: Meson not found          ← QEMU mode (소스 없이 퍼징할 때 필요, Exercise 1에서는 불필요)
[!] No cargo, please install Rust  ← Unicorn mode (불필요)
```

#### clang-18 호환성 문제 해결

clang-18은 AFL++의 일부 옵션(`-fno-if-conversion`)을 지원하지 않아 `afl-clang-fast` 실행 시 오류가 발생할 수 있다. 이 경우 `apt`로 재설치하면 clang-18 환경에 맞게 패키징된 버전이 설치된다:

```bash
sudo apt install afl++
afl-fuzz --version  # 설치 확인
```

최종적으로 **clang-18 + AFL++ 4.41a** 조합으로 진행했다.

## 계측 빌드 (Instrumentation Build)

일반 빌드된 Xpdf는 AFL++가 내부를 관찰할 수 없다. AFL++의 컴파일러(`afl-clang-fast`)로 다시 빌드하면, 코드 곳곳에 **"여기 실행됐다"는 추적 코드**가 삽입된다.

```
일반 빌드:  PDF → pdftotext → 결과   (AFL++는 내부를 모름)
계측 빌드:  PDF → pdftotext → 결과   (AFL++가 어떤 코드가 실행됐는지 실시간 파악)
```

```bash
# 기존 빌드 삭제
rm -r $HOME/fuzzing_xpdf/install
cd $HOME/fuzzing_xpdf/xpdf-3.02/
make clean

# AFL++ 컴파일러로 재빌드
CC=$HOME/AFLplusplus/afl-clang-fast \
CXX=$HOME/AFLplusplus/afl-clang-fast++ \
./configure --prefix="$HOME/fuzzing_xpdf/install/"
make
make install
```

**명령어 설명**
- `make clean` : 이전 빌드 결과물을 삭제해 깨끗한 상태로 만든다
- `CC=... CXX=...` : C 컴파일러(`CC`)와 C++ 컴파일러(`CXX`)를 지정한다. 기본 `gcc` 대신 AFL++의 `afl-clang-fast`를 사용하도록 설정한다

계측이 제대로 됐는지 확인한다:

```bash
nm $HOME/fuzzing_xpdf/install/bin/pdftotext | grep afl | head -5
```

**명령어 설명**
- `nm [파일]` : 실행 파일에 포함된 심볼(함수명, 변수명 등) 목록을 출력한다
- `grep afl` : 출력 중 `afl`이 포함된 줄만 필터링한다
- `|` (파이프) : 앞 명령어의 출력을 뒤 명령어의 입력으로 전달한다

`__afl_` 로 시작하는 심볼이 나오면 계측 빌드 성공이다.

## 퍼징 실행

### core_pattern 설정

AFL++가 crash를 제대로 감지하려면 OS의 코어 덤프 설정을 변경해야 한다:

```bash
sudo su
echo core > /proc/sys/kernel/core_pattern
exit
```

**명령어 설명**
- `sudo su` : root 권한으로 전환한다. `/proc/sys/` 아래 파일은 root만 수정할 수 있다
- `echo core > /proc/sys/kernel/core_pattern` : crash 발생 시 코어 덤프 처리 방식을 변경한다. 기본값은 외부 유틸리티로 전달하는 방식이라 AFL++가 crash를 제때 인식 못 할 수 있다

### afl-fuzz 실행

```bash
afl-fuzz -i $HOME/fuzzing_xpdf/pdf_examples/ \
         -o $HOME/fuzzing_xpdf/out/ \
         -s 123 \
         -- $HOME/fuzzing_xpdf/install/bin/pdftotext @@ $HOME/fuzzing_xpdf/output
```

각 옵션 설명:

| 옵션 | 의미 |
|------|------|
| `-i` | seed 파일들이 있는 입력 디렉토리 |
| `-o` | AFL++가 변형된 파일을 저장할 출력 디렉토리 |
| `-s 123` | 랜덤 시드 고정 (재현성을 위해) |
| `--` | AFL++ 옵션과 타겟 프로그램 명령어를 구분하는 구분자 |
| `@@` | 변형된 입력 파일 경로가 자동으로 들어가는 자리 |

**seed(시드)란?**  
- AFL++가 변형의 기반으로 사용하는 초기 입력 파일이다. 정상적인 PDF 파일 하나를 넣으면 AFL++가 이를 조금씩 변형하면서 퍼징을 시작한다. seed가 작을수록 퍼징 속도가 빠르다.  
- 실제로 1MB짜리 PDF를 seed로 넣었더니 `exec speed`가 200/sec → 30/sec로 떨어지고 timeout이 1300개 이상 발생했다. `helloworld.pdf`(678B) 하나만 사용하는 것을 권장한다.

### AFL++ 화면 읽는 법

AFL++가 실행되면 아래 같은 화면이 뜬다

![image](/assets/images/dev/fuzzing101/exercise1/3.png)

```
total crashes : 1 (1 saved)   ← crash 발견 수
corpus count  : 1344           ← 새 코드 경로 발견 수
exec speed    : 198/sec        ← 초당 실행 횟수
total tmouts  : 4              ← timeout 발생 수
```

- **corpus count**가 늘어나면 AFL++가 새로운 코드 경로를 탐색 중인 것이다
- **exec speed**가 너무 낮으면(`slow!` 표시) seed 파일이 너무 크다는 신호다
- **total crashes**에 숫자가 생기면 `out/default/crashes/` 폴더에 crash 파일이 저장된다
- 빨간색으로 표시된 **uniq. crashes** 값을 볼 수 있는데, 이것은 발견된 고유(unique) 크래시의 개수를 나타낸다.

## Crash 분석 (Triage)

### 1. Crash 재현

```bash
# crashes 폴더 확인
ls $HOME/fuzzing_xpdf/out/default/crashes/
```
![image](/assets/images/dev/fuzzing101/exercise1/9.png)


```bash
# crash 재현 (큰따옴표 필수!)
CRASH="$HOME/fuzzing_xpdf/out/default/crashes/<파일명>"
$HOME/fuzzing_xpdf/install/bin/pdftotext "$CRASH" $HOME/fuzzing_xpdf/output
```
![image](/assets/images/dev/fuzzing101/exercise1/10.png)

**주의**: 파일 경로에 큰따옴표(`"`)를 사용해야 한다. 작은따옴표(`'`) 안에서는 `$HOME`이 변수로 해석되지 않아 파일을 찾지 못한다.

`Segmentation fault`가 출력되면 재현 성공인줄 알았지만...

### 2. 디버그 빌드

GDB로 분석하려면 **디버그 심볼이 포함된 빌드**가 필요하다. `-g -O0` 옵션을 추가해 다시 빌드한다:

```bash
rm -r $HOME/fuzzing_xpdf/install
cd $HOME/fuzzing_xpdf/xpdf-3.02/
make clean
CFLAGS="-g -O0" CXXFLAGS="-g -O0" \
./configure --prefix="$HOME/fuzzing_xpdf/install/"
make
make install
```

**명령어 설명**
- `CFLAGS="-g -O0"` : 컴파일 옵션을 설정한다. `-g`는 디버그 심볼 포함, `-O0`은 컴파일러 최적화 끄기다. 최적화를 끄면 코드가 원본 소스와 1:1 대응되어 GDB에서 정확한 파일명·줄 번호를 볼 수 있다

**Note**: 디버그 빌드는 GDB 분석용이다. 퍼징을 다시 하려면 계측 빌드로 돌아와야 한다.

### 3. GDB로 분석

**GDB(GNU Debugger)** 는 프로그램이 어디서, 왜 터졌는지 추적하는 도구다.

```bash
CRASH="$HOME/fuzzing_xpdf/out/default/crashes/<파일명>"
gdb --args $HOME/fuzzing_xpdf/install/bin/pdftotext "$CRASH" $HOME/fuzzing_xpdf/output
```

GDB 실행 후:

```
(gdb) run    ← 프로그램 실행, crash 재현
(gdb) bt     ← backtrace: 어디서 터졌는지 콜스택 출력
```

![image](/assets/images/dev/fuzzing101/exercise1/11.png)

![image](/assets/images/dev/fuzzing101/exercise1/12.png)

**backtrace(bt)란?**  
- 프로그램이 crash난 시점까지의 **함수 호출 순서**를 역순으로 보여준다. `#0`이 crash가 난 함수, `#1`이 그걸 호출한 함수, `#2`가 그 위 순서로 올라간다.

## 패치 코드 분석

함수가 어느 줄에 있는지 grep으로 먼저 확인했다:

![image](/assets/images/dev/fuzzing101/exercise1/15.png)

```bash
grep -n "buildImageStream" $HOME/fuzzing_xpdf/xpdf-3.02/xpdf/Gfx.cc
# 3873:  str = buildImageStream();
# 3890:Stream *Gfx::buildImageStream() {

grep -n "buildImageStream" $HOME/fuzzing_xpdf/xpdf-4.02/xpdf/Gfx.cc
# 4868:  str = buildImageStream(&haveLength);
# 4905:Stream *Gfx::buildImageStream(GBool *haveLength) {
```

호출부(3873, 4868)와 정의부(3890, 4905) 모두 라인 번호가 다르다.
xpdf 4.02에서 코드가 추가되면서 전체적으로 줄 번호가 밀린 것이다.
중요한 차이는 줄 번호가 아니라 **함수 시그니처**다.
3.02는 `buildImageStream()`으로 인자가 없지만,
4.02는 `buildImageStream(&haveLength)`로 길이 정보를 받아오는 포인터가 추가됐다.

### 취약한 코드 (xpdf 3.02)

```bash
$ sed -n '3860,3892p' $HOME/fuzzing_xpdf/xpdf-3.02/xpdf/Gfx.cc
```
- `sed -n '시작,끝p'` : 파일의 특정 줄 범위만 출력한다. 줄 번호를 알고 나서 해당 코드를 확인할 때 사용했다

![image](/assets/images/dev/fuzzing101/exercise1/16.png)
```cpp
// Gfx.cc - opBeginImage
str = buildImageStream();   // 스트림 길이 정보 없음

if (str) {
    doImage(NULL, str, gTrue);

    // 스트림이 유효한지 확인 없이 무조건 읽음
    c1 = str->getUndecodedStream()->getChar();
    c2 = str->getUndecodedStream()->getChar();
    while (!(c1 == 'E' && c2 == 'I') && c2 != EOF) {
        c1 = c2;
        c2 = str->getUndecodedStream()->getChar();  // 💥 null 스트림 접근 가능
    }
    delete str;
}
```

### 패치된 코드 (xpdf 4.02+)

```bash
$ sed -n '4855,4910p' $HOME/fuzzing_xpdf/xpdf-4.02/xpdf/Gfx.cc
```

함수 시그니처부터 변경됐다:

![image](/assets/images/dev/fuzzing101/exercise1/13.png)

![image](/assets/images/dev/fuzzing101/exercise1/14.png)
```cpp
// 3.02
Stream *buildImageStream();

// 4.02 — haveLength 파라미터 추가
Stream *buildImageStream(GBool *haveLength);
```

실제 처리 로직도 변경됐다:

![image](/assets/images/dev/fuzzing101/exercise1/17.png)
```cpp
// Gfx.cc - opBeginImage (4.02)
GBool haveLength;
str = buildImageStream(&haveLength);  // 스트림 길이 정보를 함께 받아옴

if (str) {
    if (!doImage(NULL, str, gTrue)) {
        delete str;

    // 길이를 알면 → 스트림 끝까지 안전하게 읽기
    } else if (haveLength) {
        while ((c1 = str->getChar()) != EOF) ;
        delete str;

    // 길이 모르면 → lookChar()로 미리 확인 후 읽기
    } else {
        c3 = str->getUndecodedStream()->lookChar();
        while (!(c1 == 'E' && c2 == 'I' && Lexer::isSpace(c3)) && c3 != EOF) {
            c1 = c2;
            c2 = str->getUndecodedStream()->getChar();
            c3 = str->getUndecodedStream()->lookChar();  // 미리 확인 후 읽음
        }
        delete str;
    }
}
```

**핵심 수정 사항**: `buildImageStream()`이 스트림 길이 정보(`haveLength`)를 함께 반환하도록 수정됐다. 길이 여부에 따라 두 가지 경로로 나눠서 안전하게 처리하도록 바뀌었다.

패치된 버전에서 같은 crash 파일이 더 이상 `Segmentation fault`를 일으키지 않는지 확인한다:

```bash
# 패치된 xpdf 설치
sudo apt install xpdf

# 같은 crash 파일로 테스트

```

![image](/assets/images/dev/fuzzing101/exercise1/18.png)

![image](/assets/images/dev/fuzzing101/exercise1/19.png)

**결과**: `Segmentation fault` 없이 `Syntax Error` 메시지만 출력되고 정상 종료된다.

## 트러블슈팅 (겪은 문제들)

### afl-clang-fast 실행 시 오류

```
[-] PROGRAM ABORT : afl-gcc/afl-clang are obsolete
```

→ AFL++ 버전과 clang 버전 불일치. `sudo apt install afl++`로 재설치한다.

### configure 실행 시 공백 오류

```
bash: ./configure--prefix=...: No such file or directory
```

→ `./configure` 뒤에 **스페이스** 필수. 복사/붙여넣기 시 공백이 사라지는 경우가 있다.

### seed 파일 크기 문제로 exec speed 저하

1MB 이상 PDF(`tracemonkey.pdf`)를 seed로 넣었더니 `exec speed`가 200/sec → 30/sec로 떨어지고 timeout이 1300개 이상 발생했다. seed 파일이 클수록 AFL++가 한 번 실행하는 데 시간이 더 걸리기 때문이다. `helloworld.pdf`(678B) 하나만 남기고 재시작하니 정상화됐다.

### core_pattern 오류

```
[-] PROGRAM ABORT : Pipe at the beginning of 'core_pattern'
```

→ `sudo su && echo core > /proc/sys/kernel/core_pattern && exit` 실행한다.

### 작은따옴표로 crash 파일 경로 지정 시 오류

```
Error: Couldn't open file '$HOME/fuzzing_xpdf/...'
```

→ 작은따옴표(`'`) 안에서는 `$HOME`이 변수로 해석되지 않는다. 반드시 큰따옴표(`"`)를 사용한다.
