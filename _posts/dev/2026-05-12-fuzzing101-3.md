## 2. LibTIFF 빌드

### 다운로드 및 설치

```bash
wget https://download.osgeo.org/libtiff/tiff-4.0.4.tar.gz
tar -xzvf tiff-4.0.4.tar.gz

cd tiff-4.0.4/
./configure --prefix="$HOME/fuzzing_tiff/install/" --disable-shared
make
make install
```

### 정상 작동 확인

```bash
$HOME/fuzzing_tiff/install/bin/tiffinfo -D -j -c -r -s -w \
  $HOME/fuzzing_tiff/tiff-4.0.4/test/images/palette-1c-1b.tiff
```

각 플래그의 역할은 다음과 같다.

| 플래그 | 역할 |
|--------|------|
| `-D` | 이미지 데이터 읽기 |
| `-j` | JPEG 압축 데이터 처리 |
| `-c` | 컬러맵 정보 출력 |
| `-r` | 행(row) 단위로 읽기 |
| `-s` | 스트립(strip) 정보 출력 |
| `-w` | 경고 메시지 출력 |

플래그를 많이 사용할수록 프로그램이 더 많은 코드 경로를 실행하게 되어 코드 커버리지가 높아지고, 숨어있는 버그를 발견할 확률도 높아진다.

## 3. 코드 커버리지 측정

코드 커버리지란 코드의 각 줄이 몇 번 실행됐는지 보여주는 지표다. 퍼저가 코드의 어느 부분까지 탐색했는지 눈으로 확인할 수 있다.


### LCOV 설치

```bash
sudo apt install lcov
```

### `--coverage` 플래그로 재빌드

코드 커버리지를 측정하려면 `--coverage` 플래그를 추가해서 다시 빌드해야 한다.

```bash
rm -r $HOME/fuzzing_tiff/install
cd $HOME/fuzzing_tiff/tiff-4.0.4/
make clean

CFLAGS="--coverage" LDFLAGS="--coverage" \
  ./configure --prefix="$HOME/fuzzing_tiff/install/" --disable-shared
make && make install
```

### 커버리지 데이터 수집

```bash
cd $HOME/fuzzing_tiff/tiff-4.0.4/

# ① 이전 기록 초기화
lcov --zerocounters --directory ./

# ② 기준점 생성 (모든 라인 = 0)
lcov --capture --initial --directory ./ \
  --output-file app.info --ignore-errors source

# ③ 프로그램 실행
$HOME/fuzzing_tiff/install/bin/tiffinfo -D -j -c -r -s -w \
  $HOME/fuzzing_tiff/tiff-4.0.4/test/images/palette-1c-1b.tiff

# ④ 커버리지 저장
lcov --no-checksum --directory ./ --capture \
  --output-file app2.info --ignore-errors source

# ⑤ HTML 리포트 생성
genhtml --highlight --legend \
  --output-directory ./html-coverage/ ./app2.info
```

각 명령어의 역할은 다음과 같다.

| 명령어 | 역할 |
|--------|------|
| `--zerocounters` | 이전에 측정한 기록을 0으로 초기화 |
| `--capture --initial` | 실행 전 스냅샷 생성 (모든 라인 = 0) |
| `tiffinfo` 실행 | 실제 프로그램 실행, 각 라인 실행 횟수 카운트 |
| `--capture` | 실행 후 스냅샷 저장 |
| `genhtml` | HTML 리포트 생성 |

`html-coverage/index.html`을 열면 어느 코드가 실행됐는지 시각적으로 확인할 수 있다.

## 4. AFL++ 퍼징

### ASan이란?

ASan(AddressSanitizer)은 메모리 오류를 실시간으로 감지해주는 도구다. 
버퍼 오버플로우, use-after-free 등의 메모리 오류를 프로그램 실행 중에 잡아낸다.

### ASan 활성화 후 재빌드

```bash
rm -r $HOME/fuzzing_tiff/install
cd $HOME/fuzzing_tiff/tiff-4.0.4/
make clean

export LLVM_CONFIG="llvm-config-11"
CC=afl-clang-lto ./configure \
  --prefix="$HOME/fuzzing_tiff/install/" --disable-shared
AFL_USE_ASAN=1 make -j4
AFL_USE_ASAN=1 make install
```

### 퍼저 실행

```bash
afl-fuzz -m none \
  -i $HOME/fuzzing_tiff/tiff-4.0.4/test/images/ \
  -o $HOME/fuzzing_tiff/out/ \
  -s 123 \
  -- $HOME/fuzzing_tiff/install/bin/tiffinfo -D -j -c -r -s -w @@
```

각 옵션의 역할은 다음과 같다.

| 옵션 | 역할 |
|------|------|
| `-m none` | ASan 사용 시 메모리 제한 해제 (필수) |
| `-i` | 시드 입력 디렉토리 |
| `-o` | 크래시 파일 저장 디렉토리 |
| `-s 123` | 고정 시드 (재현성 확보) |
| `@@` | AFL이 생성한 테스트 파일 경로로 자동 대체 |

![image](/assets/images/dev/fuzzing101/exercise3/3.png)

몇 분 후 크래시가 발견되면 `out/default/crashes/` 폴더에 저장된다.

## 6. 취약점 상세 분석

### TIFF 파일 구조

TIFF는 태그(Tag) 기반 이미지 파일 형식이다. 각 태그는 이미지의 정보(가로 크기, 세로 크기, 압축 방식 등)를 담고 있다.

```
TIFF 파일
├── 태그 256: Image Width  = 1
├── 태그 257: Image Length = 1
├── 태그 259: Compression  = SGILog
├── 태그 277: Samples/Pixel = 3
└── 태그 65030: ???  ← AFL이 삽입한 악성 태그!
```

태그 65030은 TIFF 표준에 존재하지 않는 번호다. AFL이 정상 TIFF 파일을 변조하면서 임의로 삽입한 것이다.

### 취약점 발생 과정

**① 파일 읽기 단계 (tif_dirread.c)**

```
TIFFReadDirectory()
    ↓
tag 65030 발견 (count=1, type=BYTE)
    ↓
setByteArray()
    ↓
malloc(1 * 1) = malloc(1바이트)  ← 1바이트만 할당!
```

태그의 count가 1, type이 BYTE(1바이트)이므로 딱 1바이트만 메모리에 할당된다.

**② 출력 단계 (tif_print.c)**

```
TIFFPrintDirectory()
    ↓
저장된 모든 태그를 순서대로 출력
    ↓
tag 65030도 출력 시도
    ↓
tif_print.c:127
fprintf(fd, "%s", (char *) raw_data)  ← 문제의 코드!
    ↓
"%s" = NULL(\0)이 나올 때까지 무조건 읽음
    ↓
할당된 1바이트 안에 NULL 없음
    ↓
경계 너머까지 읽기 시도 → CRASH! 💥
```

### 메모리 상태 분석

ASan의 섀도우 바이트 출력을 보면 메모리 상태를 확인할 수 있다.

```
0x502000000000: fa fa 00 00 fa fa fd fa fa fa fd fa fa fa[01]fa
                                                         ↑   ↑
                                                      할당된  Heap redzone
                                                      1바이트 (경계!)
```

| 값 | 의미 |
|----|------|
| `00` | 정상 할당된 메모리 |
| `fa` | Heap redzone (경계) |
| `fd` | 이미 해제된 메모리 |
| `[01]` | 1바이트만 사용 가능한 메모리 ← 문제의 공간 |

실제 메모리 접근 과정은 다음과 같다.

```
주소:    0x...0070      0x...0071
         ┌──────────┐   ┌──────────────
         │  [01]    │   │  fa (경계!)
         │ 할당됨   │   │  할당 안됨
         └──────────┘   └──────────────

[01] 읽음 → NULL? 아님(0x01 ≠ 0x00) → 계속 읽자
[fa] 읽음 → NULL? 아님(0xfa ≠ 0x00) → 계속 읽자
                → ASan 경계 침범 감지 → CRASH! 💥
```

`%s` 포맷은 NULL(`\0`)이 나올 때까지 무조건 읽기 때문에, 할당된 1바이트 안에 NULL이 없으면 경계를 넘어서 읽게 된다.

### PoC 코드 커버리지 측정

크래시 파일을 입력으로 커버리지를 측정하면 `tif_print.c` 127번째 줄의 실행 횟수가 2로 표시된다.

```
html-coverage/index.html
  → libtiff 폴더 클릭
    → tif_print.c 클릭

127번 줄: fprintf(fd, "%s", (char *) raw_data);  → 실행 횟수: 2
```

![image](/assets/images/dev/fuzzing101/exercise3/5.png)

실행 횟수가 2인 이유는 악성 TIFF 파일 안에 처리된 태그가 2개 있었기 때문이다. 중요한 건 숫자가 아니라 해당 줄이 실행됐다는 사실 자체다.

## 7. 패치 적용

### 취약한 코드

```c
// tif_print.c:127
// NULL 나올 때까지 읽음 → 경계 침범 가능!
fprintf(fd, "%s", (char *) raw_data);
```

### 패치된 코드

```c
// value_count 바이트만 읽고 강제 종료!
fprintf(fd, "%.*s", (int)value_count, (char *) raw_data);
```

`%.*s`는 출력할 길이를 인자로 받아 딱 그 길이만큼만 읽고 강제로 멈추는 포맷이다. `value_count`는 태그 데이터의 크기(=malloc으로 할당한 크기)이므로, 할당된 범위 안에서만 읽게 된다.

| 포맷 | 동작 | 안전 여부 |
|------|------|-----------|
| `%s` | NULL 나올 때까지 무한정 읽음 | ❌ 위험 |
| `%.*s` | 지정한 길이만큼만 읽고 강제 종료 | ✅ 안전 |

```
패치 전 (%s)
[01] → NULL 아님 → 계속
[fa] → NULL 아님 → 계속  ← 경계 침범! 💥

패치 후 (%.*s, value_count=1)
[01] → 1바이트 읽음 → 강제 종료 
```


```
Tag 65030: M   ← 크래시 없이 정상 출력!

    0: [8, 8]
```

![image](/assets/images/dev/fuzzing101/exercise3/6.png)

ASan 에러 없이 `Tag 65030: M`이 정상 출력된다. `M`은 할당된 1바이트의 값(`0x4D`)을 ASCII로 출력한 것이다.

## 전체 요약

```
① AFL이 tag 65030(count=1)을 가진 악성 TIFF 생성
        ↓
② LibTIFF가 tag 65030 데이터를 1바이트만 malloc
        ↓
③ tif_print.c:127에서 "%s"로 출력 시도
        ↓
④ 할당된 1바이트 안에 NULL 없음
        ↓
⑤ 경계 너머 메모리까지 읽기 시도
        ↓
⑥ ASan 감지 → CRASH 💥
        ↓
⑦ "%.*s"로 크기 제한하여 패치 ✅
```
