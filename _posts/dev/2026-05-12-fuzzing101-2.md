# Fuzzing101 Exercise 2 - libexif

## 목표

libexif 0.6.14에서 두 가지 취약점의 crash/PoC를 찾는 것이다.

- **CVE-2009-3895** : `exif-entry.c`의 `exif_entry_fix()` 함수에서 발생하는 힙 기반 버퍼 오버플로우
- **CVE-2012-2836** : `exif-data.c`의 `exif_data_load_data()` 함수에서 발생하는 Out-of-bounds Read

---

## 취약점 설명

### CVE-2009-3895 — 힙 기반 버퍼 오버플로우

libexif 0.6.18의 `libexif/exif-entry.c`에 있는 `exif_entry_fix` 함수에서 발생하는 취약점이다.
원격 공격자가 잘못된 EXIF 이미지를 통해 서비스 거부 또는 임의 코드 실행을 할 수 있다.

### CVE-2012-2836 — Out-of-bounds Read

libexif 0.6.21 이전 버전의 `exif-data.c`에 있는 `exif_data_load_data` 함수에서 발생하는 취약점이다.
조작된 EXIF 태그가 포함된 이미지를 통해 원격 공격자가 서비스 거부를 유발하거나 프로세스 메모리의 민감한 정보를 읽어낼 수 있다.

## 5. afl-clang-lto로 재빌드

AFL++은 바이너리를 계측(instrument)해서 커버리지를 측정한다.
이를 위해 AFL 컴파일러로 다시 빌드해야 한다.

사용할 컴파일러는 `afl-clang-lto`이다. AFL++ 공식 문서에서 권장하는 우선순위는 다음과 같다.

```
clang 11+ 있음  → afl-clang-lto  (가장 추천, collision-free)
clang 6.0+ 있음 → afl-clang-fast
gcc 5+ 있음     → afl-gcc-fast
그 외           → afl-gcc
```

우리 환경은 clang 11이 있으므로 `afl-clang-lto`를 사용한다.

libexif 재빌드

```bash
rm -r $HOME/fuzzing_libexif/install
cd $HOME/fuzzing_libexif/libexif-libexif-0_6_14-release/
make clean
export LLVM_CONFIG="llvm-config-11"
CC=afl-clang-lto ./configure --enable-shared=no --prefix="$HOME/fuzzing_libexif/install/"
make
make install
```

exif CLI 재빌드

```bash
cd $HOME/fuzzing_libexif/exif-exif-0_6_15-release
make clean
export LLVM_CONFIG="llvm-config-11"
CC=afl-clang-lto ./configure --enable-shared=no \
  --prefix="$HOME/fuzzing_libexif/install/" \
  PKG_CONFIG_PATH=$HOME/fuzzing_libexif/install/lib/pkgconfig
make
make install
```

## 6. 퍼징 시작

```bash
echo core | sudo tee /proc/sys/kernel/core_pattern

afl-fuzz -i $HOME/fuzzing_libexif/exif-samples-master/jpg/ \
         -o $HOME/fuzzing_libexif/out/ \
         -s 123 -D \
         -- $HOME/fuzzing_libexif/install/bin/exif @@
```

각 옵션의 의미는 다음과 같다.

| 옵션 | 의미 |
|------|------|
| `-i` | 시드 코퍼스 디렉토리 |
| `-o` | 결과 저장 디렉토리 |
| `-s 123` | 고정 시드 (재현 가능한 결과) |
| `-D` | Deterministic mutation 활성화 (AFL++ 4.0+ 필수) |
| `@@` | AFL이 생성한 파일을 여기에 넣어서 실행 |

AFL++ 4.0 이상부터는 `-D` 옵션을 명시해야 Deterministic mutation이 활성화된다.
이 옵션 없이 퍼징하면 랜덤 변이만 수행되어 일부 취약점을 찾지 못할 수 있다.

퍼징 결과 crash 파일들은 `$HOME/fuzzing_libexif/out/default/crashes/` 에 저장된다.


## 8. CVE-2012-2836 분석

퍼징 후 발견된 crash 파일을 GDB로 분석했다.

```bash
gdb $HOME/fuzzing_libexif/install/bin/exif
(gdb) run [crash파일경로]
(gdb) bt
```

bt 결과

```
#0  0x000055555557f1fa in exif_get_sshort (
    buf=0x5556557b0ac5 <error: Cannot access memory at address 0x5556557b0ac5>, 
    order=EXIF_BYTE_ORDER_INTEL)
    at /home/ubuntu/fuzzing_libexif/libexif-libexif-0_6_14-release/libexif/exif-utils.c:94
#1  exif_get_short (
    buf=0x5556557b0ac5 <error: Cannot access memory at address 0x5556557b0ac5>, 
    order=EXIF_BYTE_ORDER_INTEL)
    at /home/ubuntu/fuzzing_libexif/libexif-libexif-0_6_14-release/libexif/exif-utils.c:104
#2  exif_data_load_data (data=0x5555557ae9f0, d_orig=<optimized out>, 
    ds_orig=<optimized out>) at exif-data.c:819
#3  0x0000555555574b40 in exif_loader_get_data (loader=<optimized out>)
    at /home/ubuntu/fuzzing_libexif/libexif-libexif-0_6_14-release/libexif/exif-loader.c:387
#4  main (argc=<optimized out>, argv=<optimized out>) at main.c:438
```

콜스택에 `exif_data_load_data`가 보이므로 CVE-2012-2836이다.
`frame 2`로 이동해 원인을 분석한다.

```bash
(gdb) frame 2
#2  exif_data_load_data (data=0x5555557ae9f0, d_orig=<optimized out>, 
    ds_orig=<optimized out>) at exif-data.c:819
819		n = exif_get_short (d + 6 + offset, data->priv->order);
```

```bash
(gdb) info locals
```

info locals 결과

```
d      = 0x5555557b0ac0 "Exif"   ← EXIF 데이터 버퍼
ds     = 11293                    ← 버퍼 전체 크기
l = <optimized out>
len = <optimized out>
offset = 4294967295               ← 💥 0xFFFFFFFF, 비정상값!
n = <optimized out>
```

`offset = 4294967295 = 0xFFFFFFFF = uint32_t 최대값`이라는 비정상적인 값이 확인된다.

```bash
(gdb) list
```

list 결과

```c
816     if (offset + 6 + 2 > ds) {   // 검증 코드
817         return;
818     }
819     n = exif_get_short (d + 6 + offset, ...);  // 💥 여기서 터짐
```

검증 코드가 있음에도 통과된 이유는 정수형 오버플로우 때문이다.

```
offset = 4294967295 (0xFFFFFFFF)
offset + 6 + 2 = 4294967295 + 8 = 4295000003

uint32_t 최대값(4294967295) 초과
→ 정수형 오버플로우 발생 → 값이 7로 뚝 떨어짐
→ 7 > 11293? → false → 검증 통과 💥
```

offset 타입 확인

```bash
grep -n "ExifLong" libexif/exif-data.c
# 702: ExifLong offset;

grep -n "ExifLong" libexif/exif-utils.h
# 41: typedef uint32_t ExifLong;  /* 4 bytes */
```

`offset`의 타입인 `ExifLong`이 `uint32_t`로 정의되어 있어 최대값이 4294967295임을 확인했다.
공격자가 파일에 `0xFFFFFFFF`를 넣으면 그 값이 그대로 offset에 저장되고, 검증에서 오버플로우가 발생해 통과된다.

전체 공격 흐름을 정리하면 다음과 같다.

```
공격자가 조작한 EXIF 파일
    ↓
offset = 0xFFFFFFFF 비정상값 삽입
    ↓
검증 코드에서 정수형 오버플로우 발생 → 검증 통과
    ↓
d + 6 + 0xFFFFFFFF → 버퍼 훨씬 밖의 주소
    ↓
exif_get_short → exif_get_sshort → 💥 Cannot access memory
```

### 패치 (CVE-2012-2836)

공식 패치 커밋을 참고해 수정했다.
[https://github.com/libexif/libexif/commit/8ce72b7f81e61ef69b7ad5bdfeff1516c90fa361](https://github.com/libexif/libexif/commit/8ce72b7f81e61ef69b7ad5bdfeff1516c90fa361)

변경 내용은 총 4가지다.

**변경 1 — 변수 선언에 fullds 추가** (`exif-data.c` 706번 줄)

```c
// 수정 전
unsigned int ds = ds_orig, len;

// 수정 후
unsigned int ds = ds_orig, len, fullds;
```

**변경 2 — ds 크기 제한 추가** (`exif-data.c`)

JPEG APP1 섹션은 최대 64KiB이므로 ds를 0xFFFE로 제한해 이후 연산에서의 오버플로우를 방지한다.

```c
fullds = ds;
if (ds > 0xfffe)
    ds = 0xfffe;
```

**변경 3 — IFD 0 offset 검증 추가** (`exif-data.c`)

덧셈 전에 `offset > ds`를 먼저 체크해 오버플로우 없이 안전하게 검증한다.

```c
// 추가
if (offset > ds || offset + 6 + 2 > ds)
    return;
```

기존의 중복되는 검증 코드는 삭제한다.

```c
// 삭제
if (offset + 6 + 2 > ds) {
    return;
}
```

**변경 4 — IFD 1 Sanity check 수정** (`exif-data.c`)

`ds - 6` 연산에서 언더플로우가 발생할 수 있으므로 조건을 분리한다.

```c
// 수정 전
if (offset > ds - 6) {

// 수정 후
if (offset > ds || offset + 6 > ds) {
```

패치 후 재빌드

```bash
cd $HOME/fuzzing_libexif/libexif-libexif-0_6_14-release/
make && make install

cd $HOME/fuzzing_libexif/exif-exif-0_6_15-release/
make clean
CC=afl-clang-lto ./configure --enable-shared=no \
  --prefix="$HOME/fuzzing_libexif/install/" \
  PKG_CONFIG_PATH=$HOME/fuzzing_libexif/install/lib/pkgconfig
make && make install
```

패치 검증

```bash
$HOME/fuzzing_libexif/install/bin/exif [crash파일경로]
```

패치 전에는 `Segmentation fault`가 발생했지만, 패치 후에는 아래와 같이 정상 종료된다.

![image](/assets/images/dev/fuzzing101/exercise2/3.png)


## 9. CVE-2009-3895 분석

퍼징 후 발견된 crash 파일을 GDB로 분석했다.

```bash
gdb $HOME/fuzzing_libexif/install/bin/exif
(gdb) run [crash파일경로]
(gdb) bt
```

bt 결과

```
#0  exif_get_slong (b=0x5555557cd000 <error: Cannot access memory>)
    at exif-utils.c:137
#1  exif_get_srational (buf=0x5555557cd000 <error: Cannot access memory>)
    at exif-utils.c:181
#2  exif_entry_fix (e=0x5555557b0040)
    at exif-entry.c:225                  ← CVE-2009-3895
#3  fix_func (e=0x5555557b0040)
#4  exif_content_foreach_entry (...)
#5  exif_content_fix (...)
#6  exif_data_foreach_content (...)
#7  exif_data_fix (...)
#8  exif_loader_get_data (...)
#9  main (...)
```

콜스택에 `exif_entry_fix`가 보이므로 CVE-2009-3895이다.
`frame 2`로 이동해 원인을 분석한다.

```bash
(gdb) frame 2
(gdb) info args
(gdb) info locals
(gdb) print e->components
(gdb) print e->size
(gdb) print e->data
```

분석 결과

```
e->data       = 0x5555557b00a0   ← 버퍼 시작 주소
e->size       = 264              ← 실제 버퍼 크기 (264바이트)
e->components = 536870945        ← 공격자가 넣은 비정상값
i             = 14828            ← 현재 루프 인덱스
```

문제 코드는 다음과 같다.

```c
for (i = 0; i < e->components; i++) {    // components 검증 없음
    sr = exif_get_srational(e->data + i * 
        exif_format_get_size(EXIF_FORMAT_SRATIONAL), o);
```

`e->components`를 검증하지 않아 루프가 버퍼 밖을 읽는다.

```
e->size       = 264바이트         → e->data[0] ~ e->data[263] 까지만 유효
e->components = 536870945
→ 루프가 536870945번 돌아야 함
→ i = 33 이후부터 버퍼 초과
→ i = 14828일 때 Cannot access memory
```

### 패치 (CVE-2009-3895)

`exif-entry.c` 224번 줄 for 루프 바로 위에 검증을 추가한다.

```c
// 수정 전
for (i = 0; i < e->components; i++) {

// 수정 후
if (e->components > e->size / exif_format_get_size(EXIF_FORMAT_SRATIONAL))
    break;
for (i = 0; i < e->components; i++) {
```

패치 후 재빌드

```bash

make && make install

cd $HOME/fuzzing_libexif/exif-exif-0_6_15-release/
make clean
CC=afl-clang-lto ./configure --enable-shared=no \
  --prefix="$HOME/fuzzing_libexif/install/" \
  PKG_CONFIG_PATH=$HOME/fuzzing_libexif/install/lib/pkgconfig
make && make install
```

패치 검증

```bash
$HOME/fuzzing_libexif/install/bin/exif [crash파일경로]
```

![image](/assets/images/dev/fuzzing101/exercise2/4.png)


패치 전후 결과 비교는 다음과 같다.

```
패치 전: Segmentation fault

패치 후: FNumber | Invalid size of entry (264, expected 536870945 x 8).
```

`Invalid size of entry`는 검증 코드가 비정상적인 components 값을 감지하고
break로 빠져나와 경고 메시지를 출력한 것이다. Segmentation fault 없이 정상 종료됐으므로 패치 성공이다.
