## 1. 들어가며 — 왜 IDS/IPS가 필요한가?


### 기존 보안 장비의 한계

방화벽(Firewall)은 IP 주소, 포트, 프로토콜 기반으로 트래픽을 허용하거나 차단한다.
하지만 이런 규칙 기반 필터링에는 명확한 한계가 있다.

| 한계 | 설명 |
|------|------|
| 허용된 포트를 통한 공격 | 80(HTTP), 443(HTTPS)는 대부분 열려 있어 공격 트래픽이 그대로 통과 |
| 내부 망 침투 후 무력화 | 방화벽은 경계(Perimeter)만 지킨다. 내부에 들어온 공격자는 탐지 불가 |
| 패킷 내용 분석 불가 | 방화벽은 헤더만 보고 판단한다. 페이로드(실제 데이터) 안의 악성 코드는 모른다 |
| 제로데이 공격 대응 불가 | 새로운 공격 기법은 방화벽 규칙에 없어 그냥 통과 |


### 침입 공격 사례

**포트 스캔 (Port Scan)**
공격자가 본격적인 침투 전, 어떤 포트가 열려 있는지 탐색하는 행위다.
방화벽은 닫힌 포트만 차단할 뿐, 스캔 행위 자체는 탐지하지 못한다.

**브루트포스 공격 (Brute Force)**
SSH, RDP 등 허용된 포트로 수천~수만 번 로그인 시도를 반복한다.
포트 22(SSH)가 열려 있으면 방화벽은 이를 정상 트래픽으로 통과시킨다.

**악성코드 통신 (C2 Communication)**
내부 PC가 악성코드에 감염된 후, 외부 C&C(Command & Control) 서버와 통신한다.
443 포트(HTTPS)를 통해 암호화된 채로 이루어지기 때문에 방화벽은 구분할 수 없다.


### 침입 탐지/차단 시스템이 등장한 배경

위와 같은 사례를 보완하기 위해 등장한 것이 바로 **IDS(Intrusion Detection System, 침입 탐지 시스템)** 와 **IPS(Intrusion Prevention System, 침입 방지 시스템)** 다.

방화벽이 "누가 들어오는가"를 통제한다면,
IDS/IPS는 "들어온 트래픽이 무엇을 하는가"를 분석한다.


| 보안 장비 | 역할 |
|-----------|------|
| 방화벽 | 허용/차단 규칙 기반 트래픽 필터링 |
| IDS | 트래픽 분석 → 이상 탐지 → 알림 |
| IPS | 트래픽 분석 → 이상 탐지 → 즉시 차단 |

## 2. IDS란 무엇인가?

### 2-1. 정의

**IDS(Intrusion Detection System)** 는 네트워크나 시스템의 트래픽을 실시간으로 분석하여, 비정상적인 활동이나 보안 정책 위반을 **탐지하고 관리자에게 알림을 발생시키는** 보안 시스템이다.

쉽게 비유하자면 "경보를 울리는 CCTV" 와 같다. 이상 행동을 발견하면 알리지만, 직접 막지는 않는다.

- **수동적(Passive) 동작**: 트래픽을 복사해 분석하므로 실제 트래픽 흐름에 영향을 주지 않는다.
- **탐지 후 처리**: 알림, 로그 기록, 관리자 통보 등 후속 대응은 사람이 담당한다.


### 2-2. 동작 원리

```
[네트워크 트래픽]
       │
       ├─────────────────────────────────► [정상 흐름 유지]
       │
       ▼ 
  [IDS 분석 엔진]
       │
       ├─ 패턴 매칭 (서명 기반)
       ├─ 이상 행동 분석 (이상 기반)
       │
       ▼
  [위협 탐지 시]
       ├─ 알림 발생
       ├─ 로그 기록
       └─ 관리자 통보
```


### 2-3. IDS의 유형

#### NIDS (Network-based IDS)

네트워크 구간에 센서를 배치해 패킷을 수집·분석하는 방식으로 침입 여부를 판단한다.

스위치의 미러 포트(SPAN Port) 나 TAP 장비에 연결하여 모든 패킷을 수집·분석한다.
미러 포트와 TAP은 NIDS가 네트워크 트래픽 복사본을 받아 분석하기 위한 대표적인 방법이다.

- **장점**: 네트워크 전체를 단일 지점에서 모니터링 가능, 초기 구현 및 관리 쉬움 
- **단점**: 암호화된 트래픽(TLS/SSL) 분석 어려움, 호스트 상에서 수행되는 행위 탐지 불가

```
인터넷 → 라우터 → 스위치 ──────── 내부망
                    │
                    │ 
                    ▼
                  [NIDS]
```

#### HIDS (Host-based IDS)

개별 호스트(서버, PC)에 에이전트(Agent)를 설치하여 해당 시스템을 감시하는 방식이다.

- **모니터링 대상**: 시스템 로그, 파일 무결성, 프로세스 실행, 레지스트리 변경
- **장점**: 암호화 트래픽도 분석 가능(복호화 후), 호스트 내부 동작까지 세밀하게 감시
- **단점**: 에이전트 설치 및 관리 필요, 에이전트 자체가 공격 대상이 될 수 있음


보통 NIDS와 HIDS를 함께 운영해, NIDS로 네트워크 경계를 감시하고 HIDS로 중요 서버 내부를 감시한다.


### 2-4. IDS의 주요 기능 요약

| 기능 | 설명 |
|------|------|
| 트래픽 모니터링 | 실시간 패킷 캡처 및 분석 |
| 이상 탐지 | 정상 패턴에서 벗어난 활동 감지 |
| 알림 발생 | 관리자에게 SNMP Trap, Syslog 등으로 통보 |
| 로그 기록 | 모든 이벤트를 저장하여 포렌식 분석에 활용 |
| 보고서 생성 | 탐지된 위협의 통계 및 트렌드 리포트 |

## 3. IPS란 무엇인가?

### 3-1. 정의

**IPS(Intrusion Prevention System)** 는 IDS의 탐지 기능에 **차단 기능**을 더한 시스템이다. 
위협을 발견하는 즉시 해당 트래픽을 자동으로 차단, 격리, 또는 수정하여 공격이 실제 피해를 주기 전에 막는다.

- **능동적(Active) 동작**: 트래픽 경로 중간에 인라인으로 배치되어 실시간 차단
- **자동화 대응**: 관리자 개입 없이 정의된 정책에 따라 자동으로 대응


### 3-2. 동작 원리

```
[외부 트래픽]
       │
       ▼
  [IPS 분석 엔진] ◄── 인라인 배치 (트래픽이 반드시 통과)
       │
       ├─ 정상 트래픽 ──────────────────► [내부망으로 전달]
       │
       └─ 위협 탐지 시
               ├─ 패킷 Drop(차단)
               ├─ 연결 초기화(TCP Reset)
               ├─ 세션 차단
               └─ 로그 기록 + 관리자 알림
```

IPS는 인라인(Inline)으로 배치되어 네트워크 트래픽은 IPS를 거쳐야 한다. 이것이 IDS와의 가장 큰 구조적 차이다.

**주의**: IPS가 인라인 배치이므로 IPS 장애 시 전체 네트워크에 영향을 줄 수 있다. 


### 3-3. IPS의 유형

#### NIPS (Network-based IPS)

네트워크 경계에 인라인으로 배치하여 외부에서 들어오는 공격을 네트워크 레벨에서 차단한다.

```
인터넷 → 방화벽 → [NIPS] → 내부망
                  (인라인)
```

- **장점**: 네크워크 전체 트래픽을 중앙에서 관리 가능, 여러 서버/PC를 한 번에 보호 가능
- **단점**: 암호화된 트래픽 분석 제한, 트래픽 병목 현상 가능 


#### HIPS (Host-based IPS)

호스트에 설치되어 악성 프로세스 실행 차단, 레지스트리 보호, 파일 시스템 무결성 유지 등을 수행한다.

- 악성코드가 실행을 시도하면 즉시 차단
- 운영체제 취약점 공격(Exploit) 방어

- **장점**: 암호화 트래픽도 분석 가능(복호화 후), 호스트 내부 동작까지 세밀하게 감시
- **단점**: 호스트마다 설치 및 관리 필요, 에이전트 자체가 공격 대상이 될 수 있음


### 3-4. IPS의 주요 대응 액션

| 대응 액션 | 설명 |
|-----------|------|
| 패킷 드롭(Drop) | 악성 패킷을 그냥 버림 |
| TCP 리셋(Reset) | 양측에 RST(Reset) 패킷을 보내 연결 강제 종료 |
| 세션 차단(Block) | 해당 세션의 이후 트래픽을 모두 차단 |
| IP 차단(Blacklist) | 공격자 IP를 일정 시간 또는 영구 차단 |
| 격리(Quarantine) | 감염 의심 호스트를 격리 VLAN(Virtual LAN)으로 이동 |
| 트래픽 수정(Scrubbing) | 악성 부분만 제거하고 나머지는 통과 |

## 4. IDS vs IPS 비교 정리

### 핵심 차이 한 줄 요약

IDS는 "이상한 사람이 들어왔습니다" 라고 **알리고**, IPS는 "이상한 사람이 들어오려 하니 막겠습니다" 라고 **행동**한다.  


### 상세 비교표

| 항목 | IDS | IPS |
|------|-----|-----|
| **동작 방식** | 수동적 (Passive) | 능동적 (Active) |
| **네트워크 배치** | 아웃오브밴드 (Out-of-band) | 인라인 (Inline) |
| **연결 방식** | 미러 포트, TAP | 트래픽 경로 중간 삽입 |
| **공격에 대한 반응** | 알림, 로그 기록 | 알림 + 즉시 차단/격리 |
| **트래픽 영향** | 영향 없음 | 지연(Latency) 발생 가능 |
| **오탐(FP) 위험** | 낮음 (알림만 발생) | 높음 (정상 트래픽 차단 위험) |
| **장애 시 영향** | 네트워크 정상 동작 유지 | 네트워크 장애 가능 |
| **주요 목적** | 공격 후 분석, 포렌식 | 실시간 공격 차단 |
| **관리자 개입** | 필요 | 자동 (정책 기반) |
| **대표 도구** | Snort(탐지 모드), OSSEC | Snort(방지 모드), Suricata |


### 배치 구조 비교

**NIDS 배치 (Out-of-band)**

```
인터넷 → 방화벽 → 스위치 → 내부망
                   │
                   │ 미러 포트
                   ▼
                  IDS (분석만, 트래픽 경로 밖)
```

**NIPS 배치 (Inline)**

```
인터넷 → 방화벽 → IPS → 내부망
                (트래픽이 IPS를 반드시 통과)
```

## 5. IDS/IPS 공통 탐지 기법

IDS와 IPS의 차이는 탐지 이후의 행동에 있을 뿐, 위협을 찾아내는 방식은 아래 세 가지로 동일하다.

```
탐지 기법 (IDS/IPS 공통)    →    IDS 대응       IPS 대응
──────────────────────────────────────────────────────
서명(Signature) 기반        →    알림/로그       트래픽 차단
이상(Anomaly) 기반          →    알림/로그       연결 초기화
하이브리드(Hybrid)          →    알림/로그       격리/필터링
```


### 5-1. 서명(Signature) 기반 탐지

이미 알려진 공격 패턴(시그니처)과 트래픽을 비교하는 방식이다. 
악성코드의 특정 바이트 패턴, 특정 포트 스캔 패턴 등을 DB에 저장해두고 매칭한다.


**동작 과정**

```
1. 트래픽 수신
2. 시그니처 DB에서 패턴 로드
3. 트래픽과 패턴 비교 (패턴 매칭)
4. 일치 시 → 알림 (IDS) / 차단 (IPS)
```

**특징**

| 구분 | 내용 |
|------|------|
| 장점 | 정확도 높음, 오탐율 낮음, 처리 속도 빠름 |
| 단점 | 신규(Zero-day) 공격 탐지 불가, 시그니처 DB 지속 업데이트 필요 |
| 대응 공격 유형 | 알려진 악성코드, 포트 스캔, SQL Injection 패턴 등 |

**시그니처 예시 (Snort 규칙)**

```
alert tcp any any -> any 80 (msg:"SQL Injection Attempt"; 
content:"union select"; nocase; sid:1000001;)
```


### 5-2. 이상(Anomaly) 기반 탐지

정상 트래픽 패턴을 학습한 후, 기준에서 벗어난 이상 행동을 탐지하는 방식이다. 머신러닝이나 통계 모델을 활용한다.

**동작 과정**

```
1. 학습 단계: 정상 트래픽 패턴 수집 및 베이스라인 생성
2. 탐지 단계: 실시간 트래픽 vs 베이스라인 비교
3. 편차 계산: 통계적 임계값(Threshold) 초과 여부 판단
4. 임계값 초과 시 → 이상 행동으로 판정
```

**탐지 예시**

- 평소 1MB/s이던 트래픽이 갑자기 100MB/s로 급증
- 업무 시간(09:00~18:00)에만 접속하던 계정이 새벽 3시에 로그인
- 특정 호스트가 짧은 시간 내 수백 개의 다른 IP로 연결 시도

**특징**

| 구분 | 내용 |
|------|------|
| 장점 | 알려지지 않은 공격(Zero-day) 탐지 가능 |
| 단점 | 오탐(False Positive)율 높음, 학습 데이터 품질에 의존 |
| 핵심 | 베이스라인(정상 패턴) 품질이 탐지율을 결정 |


### 5-3. 하이브리드(Hybrid) 방식

서명 기반과 이상 기반을 결합한 방식이다. 알려진 공격은 서명 기반으로 빠르게 탐지하고, 신규 위협은 이상 탐지로 보완한다.

```
트래픽 입력
    │
    ├── [서명 기반 엔진] ── 알려진 공격 패턴과 매칭
    │          │
    │          └── 탐지 시 즉시 처리
    │
    └── [이상 기반 엔진] ── 베이스라인과 비교
               │
               └── 임계값 초과 시 처리
```

현재 대부분의 상용 IDS/IPS 제품(Snort, Suricata, 상용 NGFW 등)이 이 방식을 채택한다.

## 6. 오탐과 미탐

IDS/IPS 운영에서 가장 중요한 균형 지점이다.

### 오탐 (False Positive)

- 정상 트래픽인데 공격이라고 잘못 판단하는 것

오탐이 많으면 진짜 공격이 경고 속에 묻힐 수 있다. 또, IPS에서 오탐이 발생하면 차단까지 하기 때문에 영향이 크다.


### 미탐 (False Negative)

- 실제 공격인데 정상으로 판단해서 높치는 것

미탐이 발생하면 시스템 침해, 데이터 유출, 랜섬웨어 감염 등 보안적으로 더 위험하다.


| 구분 | IDS       | IPS          |
| -- | --------- | ------------ |
| 오탐 | 잘못된 경고 발생 | 정상 서비스 차단 가능 |
| 미탐 | 공격 놓침     | 공격 차단 실패     |


```
      민감도(Sensitivity) 높임 →
탐지율 ↑, 미탐 ↓   BUT   오탐 ↑
           ↕ 균형점 찾기
탐지율 ↓, 오탐 ↓   BUT   미탐 ↑
      민감도(Sensitivity) 낮춤 →
```

IDS/IPS는 공격 탐지 능력만큼 오탐(False Positive)과 미탐(False Negative)을 얼마나 줄이느냐가 운영 품질을 결정하는 중요한 요소이다.

## 7. 실제 활용 방법

### 7-1. 네트워크에서의 배치 위치

기업 환경에서는 방화벽(Firewall), IPS, IDS, SIEM 등을 네트워크 구조와 보안 정책에 맞게 배치하며, 아래는 대표적인 예시이다.

```
인터넷
  │
  ▼
[경계 라우터]
  │
  ▼
[방화벽]          ← 1차 방어선: 포트/프로토콜 기반 필터링
  │
  ▼
[IPS - NIPS]      ← 2차 방어선: 딥패킷 인스펙션(DPI) 기반 차단
  │
  ├──────────────────────────────────────┐
  ▼                                      ▼
[DMZ 구간]                          [내부망]
 (웹서버, 메일서버)                  (업무 PC, DB서버)
  │                                      │
  └──────── [IDS - NIDS / HIDS] ─────────┘
                  ↓
            로그 수집 서버 (SIEM)
```

SIEM(Security Information and Event Management)은 보안 로그를 수집 및 분석해서 보안 위협을 탐지하는 시스템이다. 


### 7-2. 대표 오픈소스 도구

#### Snort

가장 널리 사용되는 오픈소스 IDS/IPS다. 규칙(Rule) 기반으로 동작하며, 모드 설정에 따라 IDS 또는 IPS로 활용한다.

```bash
# IDS 모드 (탐지만)
snort -c /etc/snort/snort.conf -i eth0 -A alert_fast

# IPS 모드 (인라인 차단)
snort -c /etc/snort/snort.conf -i eth0 -Q --daq nfq
```

#### Suricata

멀티스레드 지원으로 고성능 환경에 적합한 오픈소스 IDS/IPS다. Snort 규칙과 호환되며 YAML 기반 설정을 사용한다.

```bash
# IDS 모드
suricata -c /etc/suricata/suricata.yaml -i eth0

# IPS 모드 (NFQueue 활용)
suricata -c /etc/suricata/suricata.yaml -q 0
```

#### OSSEC (HIDS)

호스트 기반 IDS로, 파일 무결성 모니터링, 로그 분석, 루트킷 탐지 기능을 제공한다.

```bash
# 설치 후 에이전트 모드 실행
/var/ossec/bin/ossec-control start
```

### 7-3. 운영 시 고려사항

**초기 도입 단계**

1. 탐지 모드(Detection Only)로 먼저 운영
2. 오탐 패턴 파악 및 예외 규칙(Whitelist) 추가
3. 충분한 튜닝 후 차단 모드(Prevention)로 전환

**지속 운영 단계**

- 시그니처 DB 정기 업데이트 (신규 취약점 대응)
- 알림 임계값 조정으로 알림 피로 최소화
- SIEM과 연동하여 통합 로그 분석 체계 구축
- 정기적인 탐지 성능 검증 (모의 침투 테스트 활용)

## 8. IDS/IPS의 한계와 보완

IDS/IPS도 만능은 아니다. 다음과 같은 한계를 인식하고 보완 체계를 구축해야 한다.

- **암호화 트래픽**: TLS 1.3 등으로 암호화된 트래픽은 분석이 어려움 → TLS 검사(Inspection) 장비 연계 필요
- **Zero-day 공격**: 아직 시그니처가 없는 신규 공격 → 이상 기반 탐지 + 위협 인텔리전스(Threat Intelligence) 활용
- **내부자 위협**: 정상 권한을 가진 내부자의 악의적 행동 → DLP, UEBA(사용자 행동 분석) 연계
- **운영 부담**: 다량의 알림 처리 → SIEM, SOAR(자동 대응 플랫폼)과 연계


완전한 보안 체계 = 방화벽 + IDS/IPS + SIEM + EDR + 위협 인텔리전스
