## 1. CodeQL이란?

CodeQL은 GitHub에서 만든 **정적 코드 분석 도구**이다.

쉽게 말하면 **코드를 데이터베이스로 변환한 뒤, SQL처럼 쿼리를 작성해서 취약점을 검색하는 도구**이다.

일반적인 정적 분석 도구가 텍스트 패턴 매칭 방식이라면, CodeQL은 코드의 **의미(Semantic)를 이해**하고 데이터가 실제로 어떻게 흘러가는지를 분석한다.


### 분석 단계

1. CodeQL 데이터베이스를 생성하여 코드를 준비한다
2. 데이터베이스에 대해 CodeQL 쿼리를 실행한다
3. 쿼리 결과를 해석한다

#### 1) 데이터베이스 생성

CodeQL은 소스 코드를 바로 분석하지 않는다. 먼저 코드를 **데이터베이스(표 형태)로 변환**하는 과정이 필요하다.

쉽게 말하면 코드를 엑셀 표처럼 정리해두는 것이다. 이렇게 해야 나중에 원하는 패턴을 빠르게 검색할 수 있다.

언어 종류에 따라 변환 방식이 다르다.

**컴파일 언어 (C, C++, Java 등)**
- 컴파일이란 사람이 작성한 코드를 컴퓨터가 실행할 수 있는 형태로 변환하는 과정이다
- CodeQL은 이 컴파일 과정을 몰래 지켜보면서 코드 정보를 수집한다
- 수집하는 정보:
  - **구문 데이터** - 코드의 구조 (if문, 함수, 변수 등)
  - **의미 데이터** - 변수의 타입, 함수 간의 관계 등

**인터프리터 언어 (Python, JavaScript 등)**
- 인터프리터 언어는 컴파일 과정이 없어서 코드를 직접 읽어서 정보를 수집한다

수집이 완료되면 모든 정보가 **CodeQL 데이터베이스**라는 하나의 폴더에 저장된다.


#### 2) 쿼리 실행

데이터베이스가 준비되면 원하는 패턴을 검색하는 **쿼리(질문)** 를 실행한다.

쿼리는 **QL**이라는 언어로 작성하는데, SQL(데이터베이스 검색 언어)과 구조가 비슷해서 익히기 어렵지 않다.

```sql
-- SQL (일반 데이터베이스 검색)
SELECT * FROM users WHERE age > 18

-- QL (코드 검색)
from FunctionCall fc
where fc.getTarget().getName() = "strcpy"
select fc
```

쿼리를 실행하는 방법은 두 가지이다.
- **VS Code CodeQL 확장** - 에디터에서 파일 우클릭으로 바로 실행
- **CodeQL CLI** - 터미널에서 명령어로 실행


#### 3) 쿼리 결과 해석

쿼리 실행이 끝나면 결과를 **소스 코드에서 어디가 문제인지** 알 수 있는 형태로 변환한다.

결과가 표시되는 방식은 쿼리 파일 상단의 메타데이터에 따라 달라진다.

- `@kind problem` - 문제가 있는 **한 위치**에 경고 메시지 표시
  - 예: `vuln.c:6:5 → 위험한 strcpy 사용`
- `@kind path-problem` - 데이터가 **어떤 경로로 흘러갔는지** 단계별로 표시
  - 예: `gets() → vulnerable() → strcpy()` 4단계 경로 시각화
- 메타데이터가 없으면 그냥 표 형식으로만 출력된다

결과는 VS Code에서 소스 코드에 바로 표시되거나, 터미널에서 **SARIF 파일**로 저장할 수 있다.

**SARIF란?** 정적 분석 결과를 저장하는 표준 파일 형식이다. GitHub에 업로드하면 취약점 위치를 자동으로 시각화해준다.

## 2. 내부 원리

### 2-1. Data Flow Graph

CodeQL은 코드의 구문 구조가 아니라 **런타임에 데이터가 실제로 흐르는 경로**를 그래프로 표현한다.

- **AST (추상 구문 트리)** - if문, 함수, 변수 등 코드의 구조를 표현
- **Data Flow Graph** - 값이 실제로 어디서 어디로 이동하는지를 표현

예를 들어 `if문`은 Data Flow 노드가 없다. if문은 값을 전달하는 게 아니라 흐름을 제어하는 역할만 하기 때문이다.


### 2-2. Local vs Global Data Flow

| 구분 | 범위 | 속도 | 언제 씀? |
|------|------|------|----------|
| **Local** | 함수 하나 안에서만 추적 | 빠름 | 대부분의 쿼리에 충분 |
| **Global** | 함수 간, 객체 속성까지 추적 | 느림 | 복잡한 취약점 분석 |


### 2-3. Taint Tracking

Taint Tracking은 **Data Flow를 확장한 개념**이다.

일반 Data Flow는 값이 그대로 전달될 때만 추적한다. 하지만 Taint Tracking은 값이 변형되더라도 오염된 데이터가 전파되면 계속 추적한다.

```
[Data Flow]
x = 위험한 입력값
y = x + 1
→ x만 위험하다고 봄 (값이 바뀌었으니까)

[Taint Tracking]
x = 위험한 입력값
y = x + 1
→ y도 위험하다고 봄 (x에서 파생됐으니까)
```

**Source** (오염 시작점) 에서 **Sink** (위험 도착점) 까지의 경로를 추적하는 것이 핵심이다.

## 4. 실습 1 - 기본 취약점 탐지

### 4-1. 취약한 C 코드 작성

`vuln.c` 파일을 작성했다. **`strcpy()`** 는 버퍼 크기를 확인하지 않아 버퍼 오버플로우를 유발할 수 있는 위험한 함수이다.

```c
#include <stdio.h>
#include <string.h>

void vulnerable(char *input) {
    char buffer[10];
    strcpy(buffer, input);  // 취약점! 크기 체크 안함
    printf("Input: %s\n", buffer);
}

int main() {
    char *evil = "AAAAAAAAAAAAAAAAAAAAAA";  // 10자 초과 입력
    vulnerable(evil);
    return 0;
}
```

**취약점 설명**

- buffer 크기 = 10바이트
- 입력값 = 22바이트
- 12바이트 초과 → 메모리 덮어씀 → 버퍼 오버플로우 발생


### 4-2. CodeQL DB 생성

CodeQL은 gcc가 컴파일하는 과정을 지켜보면서 코드 구조를 테이블로 저장한다.

```bash
codeql database create vuln-db --language=cpp --command="gcc -c vuln.c"
```

생성된 DB 구조:

```
vuln-db/
├── db-cpp/         ← 실제 DB 테이블 (함수, 변수, 호출 정보)
├── diagnostic/     ← 분석 로그
├── log/            ← 실행 로그
├── baseline-info.json
├── codeql-database.yml
└── src.zip         ← 원본 소스코드
```


### 4-3. 쿼리 작성 및 실행

`buffer-overflow.ql` 파일을 작성했다.

```ql
/**
 * @kind problem
 */

import cpp

from FunctionCall fc
where fc.getTarget().getName() = "strcpy"
select fc, "위험한 strcpy 함수 사용! 버퍼 오버플로우 가능성 있음"
```

**쿼리 설명**

- `import cpp` - C/C++ 분석 라이브러리 불러오기
- `from FunctionCall fc` - DB에 저장된 모든 함수 호출을 fc 변수로 선언
- `where fc.getTarget().getName() = "strcpy"` - 함수 이름이 strcpy인 것만 필터
- `select fc, "..."` - 탐지된 위치와 경고 메시지 출력

![image](/assets/images/blog/codeql/1.png)

```

✅ vuln.c:6:5 → strcpy(buffer, input) 위치
✅ "위험한 strcpy 함수 사용! 버퍼 오버플로우 가능성 있음"
```

## 5. 실습 2 - Taint Tracking 커스텀 쿼리

### 5-1. 취약한 C 코드 수정

Taint Tracking은 **Source에서 Sink까지의 데이터 흐름**을 추적한다. `gets()`로 입력받은 데이터가 `strcpy()`까지 흘러가는 구조로 코드를 수정했다.

```c
#include <stdio.h>
#include <string.h>

void vulnerable(char *input) {
    char buffer[10];
    strcpy(buffer, input);  // Sink (위험 도착점)
    printf("Input: %s\n", buffer);
}

int main() {
    char input[100];
    gets(input);            // Source (오염 시작점)
    vulnerable(input);
    return 0;
}
```


### 5-2. DB 재생성

코드가 변경됐으므로 DB를 다시 생성했다.

```bash
codeql database create vuln-db --language=cpp --command="gcc -c vuln.c" --overwrite
```


### 5-3. 커스텀 Taint Tracking 쿼리 작성

`custom-query2.ql` 파일을 작성했다.

```ql
/**
 * @kind path-problem
 */

import cpp
import semmle.code.cpp.dataflow.new.TaintTracking
import semmle.code.cpp.dataflow.new.DataFlow

module MyTaintConfig implements DataFlow::ConfigSig {
    predicate isSource(DataFlow::Node source) {
        exists(FunctionCall fc |
            fc.getTarget().getName() = "gets" and
            source.asExpr() = fc.getAnArgument()
        )
    }
    predicate isSink(DataFlow::Node sink) {
        exists(FunctionCall fc |
            fc.getTarget().getName() = "strcpy" and
            sink.asExpr() = fc.getAnArgument()
        )
    }
}

module MyFlow = TaintTracking::Global<MyTaintConfig>;
import MyFlow::PathGraph

from MyFlow::PathNode source, MyFlow::PathNode sink
where MyFlow::flowPath(source, sink)
select sink.getNode(), source, sink, "오염된 데이터가 strcpy까지 흘러옴"
```

**쿼리 설명**

- `isSource` - `gets()` 함수의 인자를 오염 시작점으로 정의
- `isSink` - `strcpy()` 함수의 인자를 위험 도착점으로 정의
- `MyFlow::flowPath` - Source에서 Sink까지 경로가 존재하면 탐지


### 5-4. 결과

![image](/assets/images/blog/codeql/2.png)

```
✅ 1 result 탐지 (27초 소요)
✅ Path (4 steps) - 데이터 흐름 경로 4단계
```

**탐지된 경로:**

```
1. vuln.c:12:10  →  gets(input)              ← 오염 시작
2. vuln.c:13:16  →  vulnerable(input)         ← 함수에 전달
3. vuln.c:4:23   →  void vulnerable(char *input) ← 파라미터 수신
4. vuln.c:6:20   →  strcpy(buffer, input)     ← 위험 도착
```

단순히 위험 함수의 존재 여부가 아니라, **데이터가 어떤 경로로 흘러가는지 4단계로 시각화**된다는 점이 인상적이었다.

## 6. 표준 쿼리로 전체 분석

CodeQL은 직접 쿼리를 작성하지 않아도 **표준 쿼리 세트 전체를 한 번에 실행**할 수 있다.

```bash
codeql database analyze vuln-db codeql/cpp-queries \
  --format=sarifv2.1.0 \
  --output=result.sarif
```


**결과:**

powershell에서 실행해야 한다.

```bash
Get-Content D:\codeql\code-practice\result.sarif | ConvertFrom-Json | Select-Object -ExpandProperty runs | Select-Object -ExpandProperty results
```

![image](/assets/images/blog/codeql/3.png)

```
ruleId  : cpp/dangerous-function-overflow
message : 'gets' does not guard against buffer overflow.
```

`gets()` 함수가 버퍼 오버플로우를 막지 않는다는 취약점이 탐지됐다.


**SARIF란?**

- Static Analysis Results Interchange Format
- 정적 분석 결과를 저장하는 표준 파일 포맷
- GitHub에 업로드하면 취약점을 자동으로 시각화해준다

**3가지 분석 방법 비교:**

| 방법 | 명령어/파일 | 탐지 내용 |
|------|------------|-----------|
| 직접 쿼리 | buffer-overflow.ql | strcpy 사용 위치 |
| Taint Tracking | custom-query2.ql | gets → strcpy 흐름 경로 |
| 표준 쿼리 | database analyze | gets 위험 함수 탐지 |
